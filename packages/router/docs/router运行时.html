Router 运行时

这份文档按当前 router.ts 的运行时实现整理：路由表匹配、首屏初始化、push / replace / popstate、 异步组件加载、空闲预加载，以及 hash 锚点滚动的响应式时序。

1. 运行时数据模型

下面这些字段对应当前 Router 实例在运行时的核心状态。active 是唯一真正驱动模板重渲染的响应式出口； 其余历史、预加载、hash 状态都围绕它组织。

字段 作用 是否响应式
active 当前激活的路由条目，模板用 {active.component} 渲染。 是
RouteEntry.url 完整浏览器 URL（path + search + hash），用于区分同一路由下不同 hash 历史。 否，随 entry 保存
scrollRootId 当前滚动容器的 DOM id；滚动时即时查找节点，适配 active 渲染后的最新 DOM。 否，已加入 StoreIgnoreKeys
routes 路由表，保存 import / component / meta / layout / status。 否，已加入 StoreIgnoreKeys
menus 目录嵌套菜单，主要服务导航和文档化结构。 否
stack 历史栈，记录每次 Router 自己产生或恢复的页面。 否
stackIndex 当前栈顶指针，配合 back / forward / go / popstate。 否
idleSet 尚未预加载的路由集合，用于 #preloadNext()。 否
loadingCount 当前并发加载中的数量，限制预加载并行度。 否
navId 导航版本号，用于丢弃过期导航结果。 否
activeCommitId 每次提交 active 时递增，给 hash pending 绑定当前提交。 否
hashScrollVersion 仅用于同一个 active 下的纯 hash popstate，强制触发 post effect。 否
#pendingHash 等待滚动的 hash 任务，当前实现里绑定 activeCommitId。 否
#hashEffect 持久的 post effect，只创建一次，负责消费 pending hash。 否

2. 整体架构

Router 继承 Store，真正让模板感知变化的只有 active。 路由表、历史栈和预加载集合都被刻意排除在响应式之外。

classDiagram direction LR class Store class Router { +active: RouteEntry | null +routes: RouteMap +menus: Menu[] +enterGuard +leaveGuard +maxPreload: number +pushState(url) +replaceState(url) +back() +forward() +go(delta) -#init(path) -#initBrowser() -#initIdleSet() -#initHashEffect() -#setActive(entry, hash) -#setPendingHash(hash, trigger) -#getHash(url) -#navigate(target, opts) -#loadComponent(path) -#preloadNext() -#onClick(e) -#onPopstate() } class RouteEntry { +path: string +url?: string +params: Record~string,string~ +scroll?: number +component?: Store +meta?: Record~string,any~ +layout?: Store } class RouteRecord { +import? +component? +status +promise? +params? +meta? +layout? } Router --|> Store Router o-- RouteEntry : active / stack Router o-- RouteRecord : routes Router o-- Menu : menus note for Router "StoreIgnoreKeys = ['routes', 'menus', 'stack', 'ready']"

关键约束

  • active 变化会触发模板重渲染和 hash post effect。
  • routes、menus、stack 不参与响应式派发。
  • 所有导航最终都要收敛到一次 #setActive()。
  • hash 滚动只能在 DOM 已经可查询后发生。

为什么要有 commit id

activeCommitId 解决的是“旧导航的 pending hash 被晚到的 post effect 消费”的问题。 任何不属于当前提交版本的 hash 任务都会被丢弃。

3. 初始化流程

初始化分两层：#init(path) 负责首屏路由决策与首个 active 提交；#initBrowser() 负责挂事件和建立持久 hash watcher。两者都调用 #initHashEffect()，但它是幂等的，只会真正创建一次。

4. pushState / replaceState 导航

普通导航是最完整的主路径：先做守卫，再写 history，再加载组件，最后提交 active。 hash 滚动不是同步直接做，而是随着 active 进入 post 队列后统一处理。

flowchart TD A["pushState(url) / replaceState(url)"] --> B["match(url, routes)"] B --> C{"命中路由?"} C -- "no" --> Z["return"] C -- "yes" --> D["new URL(url, location.origin)"] D --> E["target = { path: result.url, params }"] E --> F["更新 stack / stackIndex"] F --> G["#navigate(target, opts)"] G --> H["navId++"] H --> I["检查 leaveGuard(this.active)"] I --> J{"通过?"} J -- "no" --> Z J -- "yes" --> K["检查 enterGuard(target)"] K --> L{"通过?"} L -- "no" --> Z L -- "yes" --> M["保存当前页 scrollY"] M --> N["history.pushState / replaceState"] N --> O["#loadComponent(lookupKey)"] O --> P{"navId 是否过期?"} P -- "yes" --> Z P -- "no" --> Q["同步 route.component / meta / layout"] Q --> R["#setActive(target, opts.hash)"] R --> R1["activeCommitId++"] R1 --> R2["pendingHash = {hash, activeCommitId}"] R2 --> R3["this.active = target"] R3 --> S["render effect 重新执行"] S --> T["post hash effect 读取 pendingHash"] T --> U["querySelector(decoded hash)"] U --> V["scrollIntoView({ behavior: 'smooth' })"] R3 --> W["#preloadNext()"]

导航中的保护点

  • navId 防止守卫或组件加载期间有新导航覆盖旧结果。
  • 在加载完成前如果导航已经变更，旧结果直接丢弃。
  • #setActive() 统一负责提交版本和 pending hash 绑定。

hash 为什么不同步滚动

因为模板的渲染 effect 可能在微任务 / post 队列里，DOM 不是在 active = target 这一行后立即就稳定可查。 把滚动交给 post effect，才能保证查询的是最终 DOM。

5. popstate 处理

popstate 分三种：栈内普通恢复、同一条 active 的 hash-only 变化、以及外部跳转重置栈。 这里同时兼容浏览器直接改变 hash 和 Router 自己维护的 history 栈。

flowchart TD A["window popstate"] --> B["current = pathname + search + hash"] B --> C["match(current, routes)"] C --> D["normalized = result?.url || current"] D --> E["idx = stack.findLastIndex(url == current)"] E --> F{"idx === -1?"} F -- "yes" --> G["外部跳转"] G --> H["entry = { path: normalized, params, meta, layout }"] H --> I["stack = [entry]"] I --> J["stackIndex = 0"] J --> K["#setActive(entry, location.hash)"] K --> K1["activeCommitId++"] K1 --> K2["pendingHash 绑定本次提交"] K2 --> K3["this.active = entry"] K3 --> L["render / hydrate 触发"] F -- "no" --> M{"idx === stackIndex?"} M -- "yes" --> N["仅 hash 变化"] N --> O["#setPendingHash(location.hash, true)"] O --> P["hashScrollVersion++"] P --> Q["post effect 被重新派发"] Q --> R["读取当前 pendingHash 并滚动"] M -- "no" --> S["栈内恢复"] S --> T["#saveScroll(当前 stackIndex)"] T --> U["stackIndex = idx"] U --> V["entry = stack[idx]"] V --> W["#navigate(entry, { replace: true, hash, search, saveScrollIndex:false })"] W --> X["恢复 entry.scroll"]

6. hash 滚动状态机

当前实现不是“每次导航都新建一个 effect”，而是一个持久 post watcher 监听 active 与 hashScrollVersion。它消费的是“最新 pending 任务”，而不是过去的任务队列。

stateDiagram-v2 [*] --> Idle Idle --> Armed : #setActive(entry, hash) Idle --> Armed : hash-only popstate\n#setPendingHash(hash, true) Armed --> WaitingPost : active 变化 / version bump\n进入 post 队列 WaitingPost --> Consuming : post effect 开始执行 Consuming --> Scrolling : commitId 匹配且 DOM 命中 Consuming --> Discarded : commitId 不匹配 Consuming --> TopScroll : hash 为空 TopScroll --> Idle : getElementById(scrollRootId).scrollTop = 0 / window.scrollTo(0, 0) Consuming --> Discarded : 无目标节点 Scrolling --> Idle : pendingHash = null Discarded --> Idle : pendingHash = null

为什么要有 activeCommitId

同一帧内可能发生多次导航提交。activeCommitId 让 pending hash 只能消费“当前 active 对应的那一次提交”， 旧提交即使晚到也会被丢弃。

为什么要有 hashScrollVersion

同一个 active 下的 hash-only popstate 不会改变 active，但浏览器 URL 已变。 版本号只负责把同一个 post watcher 重新派发一次；当 hash 为空时，watcher 会把页面滚回顶部。

7. 异步组件加载

路由模式里真正的组件 import 被封装在 #loadComponent() 中。 它既是普通导航的一部分，也是空闲预加载的共同底层。

flowchart TD A["#loadComponent(path)"] --> B["取 route = routes[path]"] B --> C{"route.status"} C -- "loaded" --> D["直接返回 route.component"] C -- "loading" --> E["直接返回 route.promise"] C -- "error / idle" --> F{"route.import 存在?"} F -- "no" --> G["throw Error"] F -- "yes" --> H["idleSet.delete(path)"] H --> I["route.status = 'loading'"] I --> J["route.promise = route.import()"] J --> K{"Promise 成功?"} K -- "no" --> L["route.status = 'error'"] L --> M["throw err"] K -- "yes" --> N["Comp = mod.default || mod"] N --> O["route.status = 'loaded'"] O --> P["route.component = Comp"] P --> Q{"route.meta 为空且 mod.routeMeta 存在?"} Q -- "yes" --> R["route.meta = mod.routeMeta"] Q -- "no" --> S["跳过 meta 回填"] R --> T{"route.layout 为空且 mod.layout 存在?"} S --> T T -- "yes" --> U["route.layout = mod.layout"] T -- "no" --> V["跳过 layout 回填"] U --> W["finally: loadingCount--"] V --> W W --> X["#preloadNext()"]

这条路径的一个关键点是：route 状态迁移在 routes 对象中完成，active 只接收结果，不自己维护 import 生命周期。

8. 空闲预加载

空闲预加载只处理还处于 idle 的路由，遵守 maxPreload 和 loadingCount。 它不会回头重复加载已经进入 loaded / loading / error 的路由。

flowchart TD A["#preloadNext()"] --> B{"loadingCount < maxPreload 且 idleSet 非空?"} B -- "no" --> C["return"] B -- "yes" --> D["取 idleSet 的第一个 path"] D --> E["loadingCount++"] E --> F{"requestIdleCallback 可用?"} F -- "yes" --> G["requestIdleCallback(() => #loadComponent(path))"] F -- "no" --> H["setTimeout(() => #loadComponent(path), 0)"] G --> I["预加载任务开始"] H --> I I --> J["#loadComponent(path)"] J --> K["加载结束后 loadingCount--"] K --> L["再次调用 #preloadNext()"] L --> B

9. 关键不变量与检查点

active 的所有写入点

  • #init()：首屏初始化。
  • #navigate()：push / replace 导航。
  • #onPopstate()：外部跳转重置栈，或栈内恢复。

这三个点都应该通过统一的 #setActive() 或等价路径提交，才能让 hash watcher 的语义一致。 栈内记录还需要保存 url，否则 /article 与 /article#title 会被误认为同一个历史项。

hash 的所有输入来源

  • 首屏 URL 自带 hash。
  • push / replace 导航带 hash。
  • 同一 active 下的 hash-only popstate。
  • 外部跳转或手动改地址栏后的 hash。

文档对应的实现已经把这四类输入统一到同一个 post watcher。

浏览器与 DOM 的时序

  • CSR：render 先创建 DOM，再让 post watcher 读取 hash。
  • Hydrate：SSR DOM 已存在，post watcher 只负责定位与滚动。
  • 导航中的新页面 DOM 可能来自异步渲染 effect，因此不能同步查找。
  • 浏览器原生滚动恢复被设为 manual，由 Router 在当前 scrollRootId 对应节点上统一保存和恢复。

简化版执行序列

1. 路由触发：pushState / replaceState / popstate / 首屏 init
2. 解析 URL，匹配 RouteMap，准备目标 entry
3. 守卫通过后，写 history 或恢复 stack
4. 加载组件，回填 component / meta / layout
5. 通过 #setActive() 提交 active，并绑定 pending hash
6. active 触发模板渲染或 hydrate
7. post hash watcher 消费 pendingHash
8. querySelector(hash)
9. scrollIntoView({ behavior: 'smooth' })
10. 启动 #preloadNext()

10. 备注

当前实现没有 Router teardown 生命周期，因此 #hashEffect、click 监听和 popstate 监听都是长期持有的。 如果后续要支持销毁，需要把这些监听和 effect 一起释放。

这份 HTML 只是运行时行为图，不涉及插件扫描、路由生成或 SSR 注入流程。那些内容在 packages/router/docs/plugin.md 和现有 SSR 模板中已经有单独路径。
